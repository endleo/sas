# Casino

Casino CTF, powered by [Svelte](https://svelte.dev/).

## Developing

Once you've installed dependencies with `npm install` (or `pnpm install` or `yarn`), start a development server:

```sh
npm run dev

# or start the server and open the app in a new browser tab
npm run dev -- --open
```

## Dev Guide

### File Structure

This project follows the [default structure for SvelteKit projects](https://svelte.dev/docs/kit/project-structure). 

Routing is filesystem-based, with the root at `src/routes`. The special files are `+page.svelte` (the actual page, essentially a slightly beefed-up html page) and `page.server.ts` (typescript code running on the server). A `+layout.svelte` and the corresponding `+layout.server.ts` can be used for things that have to rendered on every sub-page, e.g. a navigation bar.

Any file under `src/routes` that is **not** a `+page.svelte` or `+layout.svelte` will not be accessible to the client, so it is perfectly fine to simply but additional scripts etc. right next to the page where they are used (colocation, see [here](https://svelte.dev/docs/kit/routing#Other-files)). If something needs to be accessed by many pages, it is better to place it under `/src/lib/`, which can be addressed from anywhere as `$lib/` .For an example of this, take a look at how `auth` is imported on the login page.

### Database

Since this is just a CTF, there is no point in using a sophisticated database setup. We stick with a simple SQLite database (i.e., a database file), located on the server at `./local.db`. Our ORM is [drizzle](https://svelte.dev/docs/cli/drizzle), which provides simple access to the db and integrates well with better-auth (see below).

### Authentication

Authentication is handled by [better-auth](https://svelte.dev/docs/cli/better-auth). It provides ready-made function like `signUpEmail`, which are utilized by the `login` and `signup` pages.

Upon login, a user obtains a session cookie. This cookie is validated in a [hook](https://svelte.dev/docs/kit/hooks) (`/src/hooks.server.ts`) that runs for every request, and if it is a valid session, the variables `locals.session` and `locals.user` are set. These are only available on server-side code, i.e. in `+page.server.ts`. For an example of using them, see the server code for the `play` page, which redirects users to `/` if they are not logged in.

### Note on Stitch and Tailwind

The code generated by Stitch uses tailwind by the [play cdn](https://tailwindcss.com/docs/installation/play-cdn) - since this project already includes tailwind, this is not necessary and the corresponding `<script src=...>` can be deleted when importing code from stitch.

Stitch uses an additional script with `id = tailwind-config` (usually towards the beginning of the code) to apply theming, which should ideally not be done separately for every page - the better place to put it is under `/src/routes/layout.css`. Note that using javascript to configure tailwind is technically [deprecated](https://tailwindcss.com/docs/upgrade-guide#using-a-javascript-config-file), and it is recommended to use [theme variables](https://tailwindcss.com/docs/theme) instead.

## Building & Deploying

To create a production version of this app:

```sh
npm run build
```

You can preview the production build with `npm run preview`.

> To deploy your app, you may need to install an [adapter](https://svelte.dev/docs/kit/adapters) for your target environment.
